00 · Start Here

What an "agent" actually is

Strip away the branding and an agent is three unglamorous things wired into a loop: a language model, a set of tools it's allowed to call, and code that keeps feeding the model's output back to it until the task is done.

Concretely: you send the model a prompt plus a list of tools (each one just a name, a description, and a schema for its inputs, e.g. run_sql(query: string)). The model replies either with a final answer, or with a request to call one of those tools. Your code executes that tool call for real, appends the result to the conversation, and sends the whole thing back to the model. Repeat. That loop (think → act → observe → think again) is the entire trick.

the whole discipline of "agentic systems" is infrastructure built around making this loop survive hours, not seconds

A demo agent runs that loop for thirty seconds in a notebook. A production agent runs it for hours, gets interrupted, needs to remember what it did yesterday, must not be allowed to run DROP TABLE just because a prompt-injected web page told it to, and sometimes needs to hand part of the job to a second, more specialized agent. Those five gaps between "demo loop" and "production system" are this document's table of contents, plus a sixth: the toolkit you build all of it out of.

Why two SDKs, not one

AWS Strands and Claude Code / the Claude Agent SDK solve the same five problems because every serious agent framework eventually has to. Reading them side by side is faster than reading either one alone. The contrast is what makes each design choice legible. Where a third, vendor-neutral standard exists (MCP, and increasingly A2A), it's called out in violet because it isn't "theirs": it's a wire format both sides can speak. Section 07 then does the same exercise against this repository's own code, so the concepts stop being abstract.

01 Problem

Surviving a restart

Your agent has been talking to a customer for twenty minutes when the container it's running in gets redeployed, or your laptop sleeps, or the request just times out. Does the agent wake up remembering any of it?

By default: no. Everything (the conversation, any facts the agent had figured out, any partially-completed plan) lives in a Python object in RAM. The fix is a session: a serializable snapshot of that state, written somewhere durable, keyed by an ID you control, so a new process can rebuild the exact same agent by asking for that ID.

Failure mode this prevents

Without sessions, every crash, redeploy, or horizontal scale-out event silently amnesia's the agent mid-task. Worse, in a multi-agent pipeline, it can also strand a task half-finished with no record of what already happened, leading to duplicate side effects on resume (e.g. sending the same email twice).

a crash between two turns loses nothing but the in-flight token stream
Same Problem, Two Implementations
Strands Agents SDK
SessionManager (File / S3 backed)

A hook-based manager listens for lifecycle events (message added, invocation finished) and writes a snapshot to a pluggable backend. Swap FileSessionManager for S3SessionManager with zero other code changes: same key layout, different disk.

from strands import Agent
from strands.session.file_session_manager import FileSessionManager

sm = FileSessionManager(session_id="user-42", storage_dir="./sessions")
agent = Agent(session_manager=sm)
agent("Hello!")

# new process, same session_id → full history restored
agent2 = Agent(session_manager=FileSessionManager(
    session_id="user-42", storage_dir="./sessions"))

Persisted: the conversation, the conversation-manager's own state (see Problem 02), and a separate AgentState dict for facts you want to survive independent of the raw chat log.

Extension point, not a wall: SessionManager is an abstract hook provider: a custom Redis or DynamoDB backend is fully supported, not a workaround.
Claude Code / Agent SDK
Local transcripts vs. server-side sessions

Two different products, two different answers. Claude Code (the CLI) writes a local JSONL transcript per session, tied to your working directory, resume it with claude --resume, branch it with --fork-session. The Claude Agent SDK's managed-agent surface instead keeps sessions server-side.

# Claude Agent SDK: managed session (server-side)
session = client.beta.sessions.create(
    agent=agent.id,
    environment_id=environment.id,
)
# drive it by sending events; history lives on the server
session.events.send(type="user.message", content="…")

No forking primitive on the managed side yet: a new task means a new session. Claude Code's local `--fork-session`, by contrast, is exactly the branching primitive Strands doesn't give you out of the box.

Moving target: the managed-agent API surface is newer and evolving fast, verify method names against current docs before shipping against it.
02 Problem

Not blowing up the context window

Every model has a hard ceiling on how much conversation it can see at once. An agent that runs for hours accumulates messages (including big, boring tool outputs like a 500-row query result) far faster than a human chat does.

There are two genuinely different things people lump together as "memory," and conflating them is the single most common design mistake in this space:

  • Conversation memory: the raw back-and-forth, sitting inside the context window, subject to a hard token ceiling.
  • Knowledge memory: durable facts stored outside the context window, pulled back in only when needed, immune to trimming.
Failure mode this prevents

Hit the ceiling with no plan and one of two bad things happens: the provider throws a hard error and the agent dies mid-task, or a naive truncation silently drops the system instructions and early decisions, and the agent starts confidently contradicting itself with no idea why.

summarizing trims the window; knowledge memory sidesteps it completely
Same Problem, Two Implementations
Strands Agents SDK
ConversationManager + AgentState / memory tools

Three swappable managers own agent.messages: SlidingWindowConversationManager (default: drops oldest messages once you exceed window_size, 40 by default; keeps tool-call/result pairs intact), SummarizingConversationManager (reactively summarizes the oldest summary_ratio (0.3 by default) via an LLM call only when the provider raises a context-overflow error, always preserving the last preserve_recent_messages, 10 by default), and NullConversationManager (no-op, for stateful models managing their own history server-side).

from strands.agent.conversation_manager import SummarizingConversationManager

manager = SummarizingConversationManager(
    summary_ratio=0.5,          # summarize the oldest half
    preserve_recent_messages=5,  # never touch the last 5
)
agent = Agent(model=model, conversation_manager=manager)

For durable facts, a separate AgentState dict survives independent of the conversation, and the companion strands-agents-tools package ships retrieval tools (mem0_memory, Bedrock Knowledge Bases, Elasticsearch/MongoDB vector search): knowledge memory the agent pulls in on demand.

Counts messages, not tokens: the sliding window is message-count-based; the real token check happens server-side at the model provider and only then triggers a reduction.
Claude Code / Agent SDK
Auto-compaction + the Memory tool

Claude Code auto-compacts as context fills: older tool output goes first, then a summarization pass, with a PreCompact hook that fires beforehand so you can force-preserve something critical. The managed Agent SDK does the same thing fully automatically, with no exposed hook: compaction there is deterministic infrastructure, not a decision point.

# Messages API: a persistent, agent-managed memory store
tools = [{"type": "memory_20250818", "name": "memory"}]
# Claude issues: view /memories, create /memories/findings.md, …
# your handler executes those against real storage (disk, DB, S3)

Knowledge memory here is the Memory tool: a client-side /memories directory Claude reads and writes explicitly. It's the direct analogue of Strands' mem0_memory. Separately, CLAUDE.md (checked-in project rules) and auto-generated MEMORY.md give Claude Code persistent context loaded fresh at every session start.

Put durable rules in CLAUDE.md, not mid-conversation: instructions given early in a long session are exactly what compaction is most likely to compress away.
03 Problem

Not letting the agent nuke your infra

The moment an agent can run shell commands or write code, you've handed an LLM (which can be manipulated by text it reads, e.g. a poisoned web page or a crafted support ticket) a way to execute arbitrary instructions on whatever machine it's running on.

The fix is isolating where tool execution happens, so the blast radius of a bad instruction is a disposable box, not your production host. There's a second, lighter-weight version of the same idea worth knowing: instead of sandboxing the process, you sandbox the output: validate that whatever the model generated is provably safe before it ever runs. Section 07 has a real, minimal example of exactly that.

Failure mode this prevents

Prompt injection → arbitrary code execution on your host → data exfiltration or lateral movement into your real infrastructure. This isn't hypothetical: any tool that shells out or evaluates code, given untrusted input anywhere in its context, is a remote-code-execution primitive by default.

a bad instruction can still run; it just can't leave the box
Same Problem, Two Implementations
Strands Agents SDK
Sandbox backends (opt in): default is unsafe

Strands' strands.sandbox module decouples tool execution from the agent process: a pluggable backend (e.g. DockerSandbox) exposes execute, execute_code, and file read/write, and passing it to Agent(sandbox=...) auto-vends sandbox_bash/sandbox_file_editor tools that run inside it.

from strands import Agent
from strands.sandbox import DockerSandbox

agent = Agent(sandbox=DockerSandbox(container="scratch-box"))
agent("List all files in the current directory")
# runs sandbox_bash *inside* the container, not on your host
The default is named for exactly this reason: omit sandbox= and you get NotASandboxLocalEnvironment: it runs with the full privileges of the host process. The separate strands-agents-tools package's shell/REPL tools instead use a human-confirmation prompt (BYPASS_TOOL_CONSENT), a weaker, non-isolating safeguard. Don't confuse the two.
Claude Code / Agent SDK
Permission modes + managed cloud sandboxes

Claude Code's default posture is permission rules, not process isolation: default prompts on edits/shell commands, plan lets Claude read and reason but never write, acceptEdits auto-accepts safe file ops, bypassPermissions skips prompts entirely. Optional OS-level sandboxing can additionally restrict Bash to allowed directories/domains. Permissions and sandbox rules both apply, together.

# settings.json: deny a class of destructive commands outright
{
  "permissions": {
    "deny": ["Bash(rm -rf *)", "Bash(git push --force*)"]
  }
}

The managed Agent SDK takes the Strands-style approach by default instead: bash, file ops, and code execution run inside Anthropic-managed (or self-hosted) cloud sandboxes, with filesystem/network isolation as the enforcement layer rather than per-command prompts.

Plan mode is a sandbox substitute for exploration: when you just need Claude to look around and reason, it needs zero write privileges to do that: force it via plan instead of trusting a permission prompt to catch every edit.
04 Problem

Getting a human to say yes before something irreversible

Sandboxing bounds the blast radius of a bad action. It does nothing for a correct action you'd still rather a person confirm: deleting the right record, sending the real email, wiring real money. That's human-in-the-loop (HIL): a deliberate pause, at one specific point, that resumes exactly where it left off once a person answers.

The naive alternatives both fail: hard-blocking the tool means the agent literally can't do part of its job; trusting the model's judgment means you're one clever prompt injection away from an unapproved action going through anyway. HIL is the third option: the agent can do it, but only after a person says so.

Failure mode this prevents

An agent with real-world write access (send, delete, pay, deploy) that has no approval gate will eventually execute one of those actions on bad reasoning, a hallucinated fact, or an injected instruction, with no human ever in the decision.

the pause is resumable: the human's answer re-enters exactly where the run stopped, nothing is replayed
Same Problem, Two Implementations
Strands Agents SDK
interrupt() / InterruptException

A hook (typically on BeforeToolCallEvent) calls event.interrupt(name, reason). With no stored answer yet, this raises InterruptException, which unwinds the run: result.stop_reason == "interrupt" and result.interrupts lists what's pending. Your app shows the reason to a human, then re-invokes the agent with the answer: the same interrupt call now returns it instead of raising, and execution resumes at that exact point.

class ToolApprovalHook(HookProvider):
    def register_hooks(self, registry, **kwargs):
        registry.add_callback(BeforeToolCallEvent, self.gate)

    def gate(self, event: BeforeToolCallEvent):
        if event.tool_use["name"] != "delete_record":
            return
        approval = event.interrupt("confirm_delete", reason="destructive action")
        if approval != "yes":
            event.cancel_tool = "not approved"

result = agent("delete record 42")
if result.stop_reason == "interrupt":
    i = result.interrupts[0]
    result = agent([{"interruptResponse": {"interruptId": i.id, "response": "yes"}}])
This works identically inside a Graph or Swarm (Problem 05): the whole flow pauses at the node that raised it, not just a single agent.
Claude Code / Agent SDK
PreToolUse hooks + permission prompts + plan mode

Claude Code's HIL layer is two things stacked: broad permission rules ("ask before any Bash command") and specific PreToolUse hooks that inspect the actual command and can deny it outright, mechanically, with no LLM judgment involved in the block itself.

#!/bin/bash
# .claude/hooks/block-destructive.sh: reads tool call as JSON on stdin
CMD=$(jq -r '.tool_input.command' < /dev/stdin)
if echo "$CMD" | grep -qE 'rm -rf|DROP TABLE'; then
  echo '{"hookSpecificOutput":{"permissionDecision":"deny",
        "permissionDecisionReason":"destructive command blocked"}}'
  exit 0
fi
exit 0

Registered under PreToolUse in settings.json with a matcher (e.g. Bash). Plan mode is HIL applied one level up: the whole exploration-then-edit sequence is split into "look" (no approval needed) and "act" (a human reviews the plan before anything executes).

The mechanism, not the model, blocks the action: a hook is a deterministic script: it doesn't ask the LLM whether a command looks safe, it pattern-matches and refuses. That's the entire point.
05 Problem

Coordinating more than one agent

A single agent with fifty tools and a kitchen-sink prompt gets worse, not better, at every one of them. Past a certain complexity, splitting work across specialized agents (one that only does SQL, one that only writes reports) outperforms one generalist trying to do everything.

Failure mode this prevents

A single overloaded agent silently does a mediocre job across the board: hard to detect, because it still "works," just worse. Specialization also bounds each agent's tool access, which is a security property, not just a quality one (a report-writer agent shouldn't hold delete permissions on the database).

order is the point; wave-executes any nodes whose deps are ready
ends when an agent replies without handing off

MCP vs. A2A, in one line: MCP connects an agent to tools and data (a database, an API, a filesystem). A2A connects an agent to another agent. Use MCP for "give my agent a new capability." Reach for A2A only when two independently-built agents (possibly from different companies) need to hand work to each other.

06 Problem

Building it out of reusable parts, not one giant prompt

Everything above (sessions, compaction, sandboxes, HIL gates, orchestration) needs to be composed without turning into an unmaintainable pile of one-off glue code. Claude's ecosystem names four specific building blocks for exactly this. Strands solves the same composition problem through tools and hooks, with less of a formal packaging layer around them.

the subagent example below is what this whole diagram compiles down to
Building blockWhat it isWhy it exists
Hooks Shell commands wired to lifecycle events (PreToolUse, PostToolUse, SessionStart, PreCompact, …), configured in settings.json. Deterministic guardrails: a script either allows or denies an action; it never "hopes the model behaves." This is the actual machinery behind Problem 04.
Skills A markdown file (SKILL.md) with a name, a trigger description, and a procedure, plus optional scripts/reference files. Loads into context only when invoked. Write a multi-step procedure once (e.g. "how we deploy," "how we review a PR"), reuse it across every session without re-explaining it or paying its full token cost until it's actually needed.
Subagents A markdown file (.claude/agents/name.md) with a restricted tool list, a dedicated system prompt, and its own context window. The packaging half of Problem 05: a specialist you can hand a task to, whose intermediate work never floods your main conversation.
Plugins A folder bundling any mix of skills, subagents, hooks, and MCP servers under one plugin.json manifest. Distribute a whole capability as one install/update/disable unit, instead of scattering its parts across settings files and hoping they stay in sync.
MCP servers An external process exposing tools/resources/prompts over a standard protocol (stdio or HTTP), configured once and available to any MCP-compatible client. Write a tool integration once, use it from Claude Code, Claude Desktop, a Strands agent, or anything else that speaks MCP, instead of re-implementing "connect to Salesforce" inside every framework separately. 
# a minimal subagent: restricted tools, isolated context
---
name: sql-reader
description: Answers questions by running read-only SQL. Use for data questions.
tools: Read, Bash(psql --readonly*)
model: sonnet
---
You are a data analyst. Run only SELECT queries. Never modify data.

Notice the shape of that last example: it's Problem 03 (sandboxing, via the restricted tool list), Problem 05 (orchestration, via being a subagent at all), and a form of Problem 04 (a read-only agent structurally can't cause the class of harm HIL exists to catch): solved together, in eight lines, because the building blocks compose. That composability is the actual payoff of learning this toolkit instead of hand-rolling each piece.

07 · Applied, In This Repo

Foundry‑Lite: the same six problems, solved without an agent loop

Foundry-Lite (this repo, built for EPAM's "Being AI-Native Hackathon 2026," BIA track) is a live ops board: an LLM drafts a semantic layer over uploaded CSVs, a human approves every proposed term, natural-language questions get answered with approved, lineage-traced SQL, and insights become Jira tickets, again gated by approval.

The interesting design choice, worth calling out before anything else: backend/app/agents.py explicitly does not use an autonomous tool-calling agent loop (its docstring records that strands-agents tool-calling was probed and rejected in probe_llm.py). Instead it's a deterministic Python pipeline with one structured, schema-locked LLM call per stage via BAML. That's a legitimate sixth answer to "how do I build this": skip the loop, keep the LLM calls single-shot and typed, and get reproducibility the free-form agent-loop approach has to work much harder for.

both approval gates are the same primitive as Problem 04, applied twice
Playbook problemHow Foundry‑Lite actually solves it
01 · Sessions A Workflow object in backend/app/main.py (id, status, run_ids[]) scopes a set of related asks under one _active_workflow_id, the closest analog here to a chat "session": a container, not a raw transcript.
02 · Memory / context budget No cross-turn conversation memory at all: each stage is one structured completion. Instead, agents.py hard-caps DuckDB sample rows (≤10/table) and truncates cell values when building the schema prompt, to stay inside the model's context window. Durable state lives outside the model entirely: ontology.yaml (merged by ID, via _merge_bucket, never clobbers an already-approved term) and the full events.jsonl log.
03 · Sandboxing Real, in-repo code. guard_sql() in backend/app/agents.py is a regex allowlist that only accepts a single SELECT/WITH statement and rejects any DDL/DML keyword, and every query additionally runs against duckdb.connect(path, read_only=True), pre-checked with EXPLAIN. This is "sandbox the output" (Problem 03's lighter variant) rather than "sandbox the process." It's a genuinely good minimal pattern to reuse anywhere an LLM emits a query language instead of shell/code.
04 · Human-in-the-loop The project's actual thesis. Every ontology term and action proposal starts status="proposed", firing an approval_required event; nothing is used (queries only read status="approved" terms) or pushed to Jira until POST /api/approvals/{subject_id} resolves it and emits approval_resolved. Low-confidence joins (<0.9) and every LLM-proposed metric are forced through this gate regardless of the model's own confidence score.
05 · Multi-agent orchestration Three sequential flows: run_ontology_draft → run_ask → run_action, each an async FastAPI background task publishing to a shared EventBus. It's orchestration without agent autonomy: a fixed Python pipeline, not a Graph/Swarm/subagent-dispatch decision made by a model. A legitimate, simpler point on the same spectrum as Problem 05.
06 · Toolkit The product itself uses none of hooks/skills/subagents/MCP: every LLM call is a typed BAML function (DraftOntologyMetrics, AskQuestion, DraftActionTicket, InterpretQueryResult) with a TwoRetries backoff policy. BAML is this project's version of "don't hand-roll it": a schema-locked function call is composable and testable the same way a Claude subagent's restricted tool list is.
The actual lesson from this repo

You don't need an autonomous agent loop to need agentic infrastructure. Foundry-Lite has no while True: think, act, observe anywhere in it, and it still had to solve context budgeting, output sandboxing, and human approval gates, because those are properties of "an LLM producing consequential output," not of "an LLM in a loop." That's the generalizable takeaway to carry into your next project, agent-loop or not.

Synthesis

Which one do I actually reach for?

Neither SDK is strictly better: they optimize for different starting points. Here's the honest version.

You're already deep in AWS (Bedrock models, Lambda, existing IAM/VPC boundaries) and want the agent to feel like just another AWS service.

Strands

You want the fastest path from "idea" to "agent editing files and running commands on my machine," with an interactive dev loop (hooks, skills, subagents) you can iterate on by hand.

Claude Code

You're building a product feature (an agent embedded in your own app, not a CLI tool) and want managed session/sandbox infrastructure rather than owning that yourself.

Claude Agent SDK

You need a deterministic multi-step pipeline where the execution order is the whole point (research → analyze → report), not left to a model's judgment.

Strands Graph

Your contract needs a precise, reproducible event sequence more than it needs autonomy, like this repo.

Skip the loop: BAML/typed calls + a hand-orchestrated pipeline

You need to integrate a tool once and have it usable from more than one of the above, or from an agent your team didn't build.

MCP, either way
Reference

Glossary, in plain terms

Context window
The maximum amount of conversation + tool output a model can "see" at once, measured in tokens. Fixed per model, not per your patience.
Tool / function call
A capability you expose to the model (a name, a description, and an input schema) that it can request be executed on its behalf.
Session
A durable, resumable snapshot of an agent's conversation and state, keyed by an ID, so a restart doesn't mean starting over.
Compaction
Automatically shrinking old conversation history (by dropping or summarizing it) so a long-running agent doesn't exceed the context window.
Sandbox
An isolated execution environment (typically a container) where an agent's shell/code tools run, so a bad instruction can't reach your real infrastructure. Guarding the LLM's output (like a SQL allowlist) is a lighter-weight cousin of the same idea.
Human-in-the-loop (HIL)
A deliberate pause point where a real person must approve before the agent proceeds with a specific, usually irreversible, action.
Hook
A script wired to a specific moment in an agent's execution (before a tool runs, after a session starts, …) that can observe, modify, or block what happens next.
Subagent
A specialized agent with its own restricted tools and context window, dispatched to by a main agent for a bounded piece of work.
MCP (Model Context Protocol)
An open, vendor-neutral standard for connecting an agent to external tools and data sources, so the integration is written once and reusable anywhere.
A2A (Agent2Agent)
An open, vendor-neutral standard for two independently-built agents to communicate: the "MCP, but for talking to other agents" layer.
Graph / Swarm (Strands)
Two multi-agent patterns: Graph is a fixed, ordered pipeline; Swarm lets agents decide amongst themselves who handles what next.
BAML (this repo)
A typed-function framework for LLM calls: you declare the input/output schema, BAML generates a client that guarantees the model's output parses into it, or retries.
